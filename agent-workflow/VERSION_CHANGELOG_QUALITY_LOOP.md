# 版本、变更日志、Git 与缺陷测试闭环（约束）

本文约定：**如何把「改了什么」存档、如何编号版本、如何提交 Git、如何记 Bug、如何从 Bug 长出测试用例并复测**，以及与 **评审、库表、Hotfix** 的衔接。与 [`AICODING_WORKFLOW.md`](./AICODING_WORKFLOW.md) 并行：前者偏协作节奏，本文偏**交付留痕与质量门禁**。

**「自动」说明：** 下文规定**不得省略的环节**；一键 bump/tag 由子项目脚本或 CI 实现（如 Changesets、`standard-version`、`mvn release`）。**栈确定后把命令写进子项目 README / 本包 [`CLAUDE.md`](./CLAUDE.md)「构建与测试命令」。**

**追踪真源：** **以 [`REPOSITORY.md`](./REPOSITORY.md)「团队真源」为准**（Issue 系统、版本字段、CI、默认分支、本地命令；根目录 `README.md` 仅入口）。本文不再重复填写，避免两处漂移。

---

## 0. 交付闭环定义

每个大需求和每个 AT-T 的完成状态，必须同时满足四项：

| 目标 | 检查口径 |
|------|----------|
| **完整性** | REQ / DSL / Plan / ATOMIC / 测试计划覆盖需求描述、页面结构、交互行为、事件、联动边界、验收标准和非目标。 |
| **可追溯性** | 代码 diff、Bug、测试、CHANGELOG、Git 提交能反查到 REQ / DSL / Plan / AT-T；口头结论不算追踪证据。 |
| **可维护性** | 工程规范、成熟方案选择、必要注释、测试、项目文件索引与实现保持一致，方便后续修改。 |
| **可交接性** | Handoff / Memory / 验证证据 / 遗留风险 / 下一步 / 提交与版本记录状态足够让新会话或人类工程师接手。 |

**缺口处理：** 任何一项缺失时，不得宣布完成；要么补齐证据，要么在 REQ / Bug / Handoff 中写明例外、责任人、风险和后续动作。

## 1. 日志存档（变更可追溯）

| 载体 | 用途 |
|------|------|
| **`CHANGELOG.md`**（正文：**[`CHANGELOG.md`](./CHANGELOG.md)**；根目录另有入口） | **对外**可读：行为变化、修复、破坏性变更、迁移提示。 |
| **Git 提交历史** | **对内**原子变更；与 Issue/PR 交叉引用（§4）。 |
| **Issue**（与 **团队真源** 所选系统一致） | Bug/需求讨论、复现、结论、验证版本。 |

**原则：** 对外「这一版做了什么」以 **`CHANGELOG.md` + 版本号/tag** 为准；口头与即时通讯不算存档。

**CHANGELOG 推荐结构**（与 [Keep a Changelog](https://keepachangelog.com/) 兼容）：

```markdown
## [Unreleased]

### Added
### Changed
### Fixed
### Removed

## [1.2.0] - 2026-05-09
...
```

---

## 2. CHANGELOG 何时必须更新（触发规则）

| 变更类型 | `[Unreleased]` |
|----------|----------------|
| 用户可见行为（UI、API、默认值、权限） | **必须**，选 Added/Changed/Fixed/Removed |
| Bug 修复且对外可见 | **必须** `Fixed` |
| **破坏性变更** | **必须** + 迁移说明或文档链接 |
| 依赖升级（无行为变化） | `Changed` 一行简述或「chore: 依赖升级」类提交说明；是否写入 CHANGELOG 由团队统一（建议写一条便于运维感知） |
| 纯重构、无行为变化 | `Changed` 注明「内部重构，行为不变」或团队约定可省略 |
| 仅文档、注释、私有脚本 | 可省略；若对外 README 行为描述变了则 **必须** |

**模糊时：** 默认 **写一条**，宁可短也不要漏。

## 2.1 分支与发布默认策略（可与团队约定微调）

| 约定 | 说明 |
|------|------|
| **日常开发** | 短生命周期分支 → PR → 合并进默认分支（`main` / `master`，以 README 为准）。 |
| **版本标签** | 对外发布对应 **tag**（如 `v1.2.0`），与权威版本字段一致（§3）。 |
| **可选 release 分支** | 需冻结测试窗口时，可从某 commit 拉 `release/x.y`，仅收 bugfix，再合并回主干并打 tag。 |
| **Hotfix** | 仍遵循 §6（从稳定 tag/分支拉出，PATCH bump）。 |

**禁止：** 仅在即时通讯宣布发版而不更新 `CHANGELOG.md` + tag（除非团队明确采用另一种对外口径并书面约定）。

---

## 3. 版本号（必须一致、可指向）

- **格式：** SemVer `MAJOR.MINOR.PATCH`；预发布如 `1.2.0-beta.1`（团队统一即可）。
- **单一真源：** 每个**可独立发布的产物**一条权威版本（见 §9 Monorepo）。
- **发布最小三步：**  
  1. `[Unreleased]` → 归档到 `[x.y.z] - YYYY-MM-DD`  
  2. bump 权威字段  
  3. `git tag v{x.y.z}`（或约定前缀）指向发布合并提交  
- **CI：** 若有流水线，建议校验 **tag 与权威版本一致**。

---

## 4. Git 提交（强制环节）

- 格式：`CLAUDE.md` 约定 `<type>(<scope>): <description>`。
- **关联 Issue：** footer 带追踪 ID（与 **团队真源**「Issue ID 写法」一致，见 [`REPOSITORY.md`](./REPOSITORY.md)）。
- **粒度：** 一提交一事；大功能多提交，release 时由 CHANGELOG 汇总。
- **禁止：** 长期大块未提交；至少按日推送分支或提交（团队可收紧）。

---

## 5. 库表变更与发布顺序（与 `CLAUDE.md` 对齐）

涉及 Schema 时：

1. **迁移脚本入库**（Flyway / Liquibase / 约定目录），附 **回滚或不可逆说明**（见 `CLAUDE.md`「数据库变更」）。
2. PR 描述或 CHANGELOG 写明：**部署顺序**（先迁移再发代码 / 或兼容双写窗口），避免无人知晓的依赖。
3. 对外若可见（新字段必填等），**破坏性**按 §2 标出。

---

## 6. Hotfix 与回滚（收窄流程）

**Hotfix（线上紧急）：**

- 从 **tag 对应的稳定分支**拉 short-live 分支（命名如 `hotfix/x.y.z+1`）。
- **最小 diff** 修一个问题；CHANGELOG `[Unreleased]` 立即补 `Fixed`。
- **PATCH** 版本 bump + tag；合并回主干时带上同一提交（或 cherry-pick），避免丢修复。
- **测试：** 至少跑 CI 规定子集；并满足 **§11** 的最小真实环境验证（或按 §13 登记债务与截止日）。

**回滚：**

- **代码回滚：** revert 发布合并提交或重置到上一 tag（按团队 Git 策略）。
- **数据库：** 已在 `CLAUDE.md` 强调不可逆变更风险；回滚方案须在迁移评审时考虑过。

---

## 7. 人工评审与 UAT

- **Code Review：** 默认 **必填**（一人即可的小仓库可写在「团队填空」里豁免规则）。关注：是否超范围、安全敏感点、与 CHANGELOG 是否一致。
- **UAT / 产品验收：** 对业务规则强依赖的功能，在 Issue 或 PR 勾选「已 UAT」并注明验收人；未通过不得合并发布分支。

---

## 8. CI 与门禁（诚实约定）

- **理想：** PR 必须通过 lint + test + build；禁止跳过着色合并（branch protection）。
- **若无 CI：** 以 PR 模板自检 + Reviewer 本地跑命令为门禁；**风险自担**，应尽快补最小流水线。

---

## 9. Monorepo / 多包

- **各自可发布的包**（多个 `package.json` / 多个 Maven module）：各自维护 **CHANGELOG 片段或独立 CHANGELOG**，各自 bump；根 CHANGELOG 可做索引或汇总。
- **单一产物**（仅根版本有意义）：只维护 **根** `CHANGELOG.md` + 单一权威版本字段。

---

## 10. 记录 Bug（Issue 模板）

字段至少包含（可与 `.github/ISSUE_TEMPLATE` 同步）：

```text
环境：版本/commit 或 tag、浏览器/OS、角色（若有关）
复现步骤：1. … 2. … 3. …
期望行为：…
实际行为：…
日志/截图：（脱敏）
是否回归：（曾正常的版本）
```

**修复后结论区：** 根因、关联 PR/commit、**首次包含修复的版本号/tag**。

## 11. 真实环境与正式用例（功能交付默认强制）

**目的：** 禁止用「过于简化的沙盒」冒充已完成的功能；每个功能必须在**接近生产的真实运行形态**下被验证。

### 11.1 什么叫「简易沙盒」（不足以单独作为交付证据）

- 仅用内存 mock、完全不拉起真实浏览器/真实后端进程、不访问约定集成环境的「玩具」验证。
- 纯 UI 层 **JSDOM / 纯组件快照** 若**未**配合真实浏览器或真实应用壳，不得作为**唯一**完成依据。
- 后端仅靠单测里全面 Mock 掉数据库与其它服务，但**没有任何**「真实进程 + 真实依赖（或 Testcontainers 等同级别）」下的集成验证。

（单测、快照、mock **允许且鼓励**作为快速反馈；但它们**不能取代**本节要求的真实环境层级证据。）

### 11.2 每个完成的功能必须同时具备

1. **书面测试用例（全面）**  
   覆盖：主路径、主要分支、权限/角色差异、错误与边界、回归关注点。  
   载体任选其一或组合：**Issue / PR 正文**、`docs/quality/test-plans/<功能简述>.md`、或自动化用例的 **describe/标题** 能映射回上述条目（评审时可逐条勾选）。

2. **真实环境验证**  
   - **浏览器交付**（Web）：在 **真实浏览器** 中执行用例——自动化须使用 **Playwright / Cypress / WebDriver** 等驱动真实浏览器（headed/headless 均可）；或经批准的 **手工步骤表 + 执行记录**（截图/录屏摘要，脱敏）。  
   - **非浏览器应用**（桌面 / 移动 / 小程序等）：在 **目标运行时或官方模拟器/真机** 上按同一套用例执行。  
   - **纯后端 API**：在 **真实 Spring Boot（或等价）进程** + **真实数据库或团队认可的集成替身（如 Testcontainers）** 上执行集成/契约级用例；禁止仅靠「全 Mock 的单测」宣称接口行为已验收。

3. **可追溯证据进 PR**  
   CI 中 E2E/集成报告链接、或手工 checklist 勾选说明、或录屏/截图索引（**脱敏**）。

### 11.3 Bug 修复

须能在 **§12** 流程下复测；复测环境层级**不得低于**该 Bug 原始发生环境（浏览器/应用/集成层），除非 Issue 中记录理由并获批。

---

## 12. Bug → 测试 → 修复 → 复测

| 步骤 | 约束 |
|------|------|
| 冻结复现 | 稳定最小步骤；可附脚本/录屏链接 |
| 测试先行 | 自动化可覆盖层：**先失败用例再改代码**（§13 例外）；且符合 **§11** 真实环境证据要求 |
| 修复提交 | 引用 Issue；CHANGELOG `Fixed`（§2） |
| 复测 | §14 清单；非作者或 CI 执行自动化部分 |
| 关闭 Issue | 写明验证版本/tag；复发则 reopen |

---

## 13. 「先写失败测试」与真实验收的例外

避免与 Hotfix/极小改动刚性冲突；**例外不等于免掉 §11**：最多延后或收窄证据形式，须在 Issue/PR **登记债务与截止日**。

| 情形 | 做法 |
|------|------|
| **Hotfix** 且窗口极短 | 最小化真实环境验证（如 staging 一条核心路径）+ Issue 登记「补全用例与 E2E 截止：____」 |
| **一行 obvious fix** | PR 说明理由；仍须在真实环境做 **回归抽样**（§11.2），或登记技术债 |
| **自动化暂时无法实现** | **手工全面用例表** + 执行记录必须进 PR；同时登记自动化补齐期限 |

**禁止：** 以「E2E 太贵」为由长期只做单元测、从不做真实浏览器/集成验证——成本问题应通过分期自动化或手工表解决，而不是删除 §11。

---

## 14. 复测清单（合并 / 发布前）

- [ ] **自动化：** lint / test / build（命令见 `CLAUDE.md` 或子项目 README）
- [ ] **§11 真实环境：** 本 PR 涉及的功能或 Bug，已在浏览器/目标应用或集成环境中按**书面用例**验证；证据已在 PR 可访问（报告/勾选表/脱敏截图索引）
- [ ] **CHANGELOG：** 符合 §2
- [ ] **版本：** 发布合并时权威字段与 tag 对齐 §3
- [ ] **Bugfix：** Issue 复现步骤在目标构建上不可复现；测试或手工步骤已落地
- [ ] **契约：** API/配置文档已更新（若 §2 要求对外可见）
- [ ] **回归抽样：** 相邻核心路径在真实环境中抽检（按风险）
- [ ] **Review / UAT：** 符合 §7
