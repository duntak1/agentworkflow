1. 这是什么

它解决什么

防止 AI 直接猜需求、乱写代码、忘记上下文、漏测试、漏 Bug、漏交接。所有关键动作都落到项目文件中，工程师可以追溯。

它不是什么

它不是后台服务，不会自动监听代码变化；也不是两个 Agent 的实时聊天系统。它是 CLI + 文档 + 流程约束。

核心命令

所有工具共用 ./scripts/aw。Codex、Cursor、Claude、Copilot、Windsurf、Cline 都按同一套项目文件协作。

最重要的规则：DSL 没有工程师确认成 已审 前，不写业务代码；每个 AT-T 开始前，Agent 必须先问清需求并等待工程师确认。

2. 工程师与 Agent 分工

角色负责什么不能省略的动作
工程师 / 产品负责人提供需求、确认 DSL、确认计划、确认子任务边界、确认高风险操作和 Git 提交。确认关键业务规则、权限、金额、数据口径、接口契约、发布风险。
Codex / Agent读取项目文件，生成 DSL/Plan，拆任务，执行代码修改，跑验证，记录 Bug/变更/Handoff。不得猜需求；不得越过 DSL/Plan/AT-T；不得自动 Git 提交。
Reviewer Agent只读代码并记录评审，不直接修改业务代码。使用 aw agents review 留痕，阻塞项必须写清楚。
Tester Agent维护测试计划、验证命令、复测和失败记录。测试失败必须进入 AI_BUG_LOG.md。

在业务项目里安装

cd /path/to/your-project
chmod +x scripts/aw scripts/*.sh
./scripts/aw init
./scripts/aw status
./scripts/aw dashboard

配置代码托管平台与构建目标

# 先询问工程师使用哪种代码托管平台
# 1=GitHub，2=本地 Git，3=GitLab，4=Bitbucket，5=Gitee，6=GitCode
# 7=Gitea，8=Forgejo，9=GitLab CE，10=Gerrit，11=云效 Codeup
./scripts/aw config init --project-kind 1 --repo-url https://github.com/owner/repo

# 本地 Git 仓库不需要远程地址
./scripts/aw config init --project-kind 2

# 构建目标：1=前端，2=后端，3=前后端
./scripts/aw config init --build-target 1
注意：这些配置不是装饰信息。后续 Plan 拆分、验证命令、远程仓库引导、任务执行都会读取 docs/PROJECT_CONFIG.md。

4. 按场景选择路线

开始真实项目之前，先让工程师选择项目形态。不同形态的目录、Agent 角色、同步方式和提示词不同，不能混用。

项目场景推荐目录Agent 会做什么工程师怎么启动
仅前端项目project-frontend/只生成前端 DSL/Plan/AT-T，重点确认页面、路由、状态、接口契约、组件和验收。使用 19.6 提示词，构建目标选 1=前端项目。
仅后端项目project-backend/只生成后端 DSL/Plan/AT-T，重点确认接口、权限、错误码、事务、数据表、消息和验收。使用 19.8 提示词，构建目标选 2=后端项目。
同电脑前后端两个项目project-frontend/
project-backend/
project-harness/		前后端各自开发，各自维护本项目事实，通过本地同步中心交换 handoff、REQ、Bug、接口契约。使用 19.10 提示词，分别在两个项目会话中声明项目路径和同步中心路径。
不同电脑前后端两个项目两端各有本地项目；project-harness 是单独远程 Git 仓库。每端先拉取 远程 Harness，再 aw sync pull、aw sync inbox、aw sync board；跨端事项用 aw sync event，经确认后提交 Harness。使用 19.11 提示词，先准备 远程 Harness 仓库。
选择原则：如果前端和后端有独立仓库、独立启动命令或不同工程师维护，就按两个项目处理；如果它们在一个 monorepo 中统一构建，可以按一个前后端项目处理，但 Plan 仍要拆出 frontend/backend 两类 AT-T。

7. Reference 与 DSL

Reference 是输入材料。DSL 是把需求变成可审阅规格的中间层。复杂项目建议使用多文件 DSL suite。

./scripts/aw dsl suite user-center "用户中心"
# 填写 docs/dsl/DSL_USER_CENTER/
# 00-requirements.md
# 10-pages.md
# 20-interactions.md
# 30-events.md
# 40-boundaries.md
# 90-acceptance.md

./scripts/aw dsl review docs/dsl/DSL_USER_CENTER/INDEX.md --write
./scripts/aw approve dsl docs/dsl/DSL_USER_CENTER/INDEX.md --plan
DSL 文件说明
00-requirements.md需求描述、目标用户、范围、非目标、约束。
10-pages.md页面结构、模块结构、字段、路由。
20-interactions.md用户操作、状态变化、异常态、空态。
30-events.md事件、触发条件、数据流、异步行为。
40-boundaries.md前后端联动边界、接口、权限、错误码、数据口径。
90-acceptance.md可检查验收与测试锚点。

8. Plan 与任务拆分

DSL 确认后，再按前端、后端或前后端生成 Plan 和 ATOMIC_TASKS。

# 只生成前端计划
./scripts/aw approve dsl docs/dsl/DSL_USER_CENTER/INDEX.md --plan --domain frontend

# 只生成后端计划
./scripts/aw approve dsl docs/dsl/DSL_USER_CENTER/INDEX.md --plan --domain backend

# Plan 落盘后审批
./scripts/aw approve plan docs/plans/PLAN_USER_CENTER_FRONTEND.md
./scripts/aw confirm docs/dsl/DSL_USER_CENTER/INDEX.md docs/plans/PLAN_USER_CENTER_FRONTEND.md
Confirm 的意义：只有 DSL 是已审、Plan 是可执行，才能进入业务代码阶段。它会生成工程师索引，但 Agent 不应把 ENGINEERING_INDEX.md 当上下文全文读取。

工程师索引与代码文件索引

确认 DSL 和 Plan 后，AgentWorkflow 会生成两类索引，分别服务于交付追溯和工程师手动优化代码。

文件读者作用更新方式
ENGINEERING_INDEX.md人类工程师工程交付总索引，聚合 DSL、Plan、AT-T、REQ、Bug、测试计划、Handoff、发布、追溯和 docs/FILE_INDEX.md 入口。aw confirm <dsl> <plan> 生成确认态；aw index 仅扫描刷新。写入 REQ / Bug / TP / DSL / Plan 等命令会自动尝试刷新。
docs/FILE_INDEX.md人类工程师项目代码文件索引。优先列出前端业务代码、后端业务代码、共享代码、测试代码、运行配置，方便工程师查找 AI 写过的代码并手动审查、优化、接手。aw file-index 生成；新增 / 删除 / 重命名前端、后端、共享、测试或配置代码文件后刷新。
边界：ENGINEERING_INDEX.md 和 docs/FILE_INDEX.md 都是给人类工程师看的索引文件。Agent 不应把 ENGINEERING_INDEX.md 当上下文全文读取；需要查代码时应根据任务、DSL、Plan 和工程师指令精确读取相关文件。

9. 研发阶段

每个子任务都必须按同一条生命周期执行。

aw next找下一项 AT-T
→
task brief输出需求沟通包
→
工程师确认问清边界和验收
→
task start进入编码
→
task complete验证并更新状态
./scripts/aw next
./scripts/aw task brief AT-T1-001
# 工程师确认后
./scripts/aw task confirm AT-T1-001 "已确认：范围、验收、非目标、接口字段。"
./scripts/aw task start AT-T1-001
./scripts/aw paste task
# 编码完成后
./scripts/aw task complete AT-T1-001
禁止：没有 task confirm 就开始写代码。需求不清楚时，Agent 必须继续追问，不能靠猜。

10. 测试、Bug、变更

测试计划

./scripts/aw tp new user-list "用户列表测试"
./scripts/aw tp link AT-T1-001 docs/quality/test-plans/TP-xxx.md
./scripts/aw verify --task AT-T1-001 --run-e2e

Bug 记录

./scripts/aw bug add "用户列表接口缺少 avatarUrl" \
  --source runtime \
  --scope AT-T1-001

研发中变更

./scripts/aw req change AT-T1-001 "用户状态枚举调整为 enabled|disabled" \
  --impact "影响筛选和状态展示" \
  --acceptance "enabled 显示启用，disabled 显示停用"

研发中变更会回写 REQ / DSL / Plan / ATOMIC，并重置当前任务需求确认，避免 Agent 继续基于旧需求写代码。

11. Handoff 与 Memory

功能用途命令
Handoff当前目标、进度、阻塞、下一步、风险。aw handoff "focus" --write
Memory长期复用的稳定结论、偏好、流程、风险。aw memory add
Chat Memory保存聊天摘要、决策、待办、待确认，不保存逐字全文。aw memory chat
 
./scripts/aw handoff "用户中心前端开发进度" --write
./scripts/aw handoff --check
./scripts/aw memory chat user-center-discussion "用户中心讨论" \
  --summary "讨论了用户列表字段和状态枚举" \
  --decisions "status 使用 enabled|disabled" \
  --todos "后端确认 avatarUrl 空值规则" \
  --open "权限失败错误码待确认" \
  --related "AT-T1-001"

12. 工程规范

工程规范集中在 docs/ENGINEERING_RULES.md。里面包含团队固定前端栈、后端栈、安全底线、注释原则、成熟方案优先、Git 提交流程，以及 ENGINEERING_INDEX.md / docs/FILE_INDEX.md 的维护边界。

./scripts/aw rules init
./scripts/aw rules discover --write
./scripts/aw rules review
./scripts/aw check rules
  • 前端固定栈：Vue3、Vite、TypeScript、Pinia、Vue Router、Axios、Element Plus、EleAdmin。
  • 后端固定栈：Java 8、Spring Boot 2.6.13、Spring Cloud Alibaba、MyBatis-Plus、Redis、Kafka、MinIO、Nacos 等。
  • AI 写代码必须有必要注释，但不能写废话注释。
  • 能用成熟方案、官方 SDK、框架内置能力时，不手写脆弱实现。
  • 新增、删除、重命名前端、后端、共享、测试或配置代码文件后，刷新 docs/FILE_INDEX.md，方便工程师手动优化代码。

14. Agent 团队

Agent 团队不是常驻 AI，而是角色、边界、交接和评审记录。它让工程师知道“谁负责什么、能改哪里、交给谁、是否阻塞”。

./scripts/aw agents assign \
  --role developer \
  --owner frontend-agent \
  --scope "AT-T-FE-001 用户列表页面" \
  --allowed "src/views/user,src/api/user,src/router" \
  --blocked "backend,docs/dsl" \
  --related AT-T-FE-001

./scripts/aw agents handoff \
  --from frontend-agent \
  --to backend-agent \
  --scope AT-T-FE-001 \
  --done "页面结构已完成" \
  --todo "请确认 /api/users 字段" \
  --risk "字段不一致会导致联调返工" \
  --evidence "docs/handoff/PROJECT_HANDOFF.md" \
  --related AT-T-FE-001

./scripts/aw agents gate --strict

任务锁与心跳

多 Agent 真正开始写代码前，需要先认领任务锁。长任务过程中通过 heartbeat 同步状态；完成、阻塞或转交时释放或 handoff。

./scripts/aw agents claim \
  --agent frontend-agent \
  --task AT-T-FE-001 \
  --role frontend \
  --scope "用户列表页面" \
  --allowed "src/views/user,src/api/user" \
  --ttl-min 120

./scripts/aw agents heartbeat \
  --agent frontend-agent \
  --task AT-T-FE-001 \
  --status working \
  --action "正在接入 /api/users mock 数据" \
  --next "完成表格状态展示"

./scripts/aw agents lock-check --task AT-T-FE-001
./scripts/aw agents release --agent frontend-agent --task AT-T-FE-001

15. 前后端双项目同步

当前后端在两个项目或两台电脑上时，用第三个目录或远程 Git 仓库作为同步中心。同步中心不只保存前后端快照，还应保存一份前后端共同确认的共享 DSL 和协作 Plan，避免两边各自拆分后产生歧义。

同步基线读取 global DSL / Plan / TASK_BOARD
→
拉取收件箱sync pull + sync inbox
→
事件发布sync event 记录 complete/change/contract 等
→
人工确认 Git工程师确认后提交 project-harness
→
对方采纳pull/inbox/board 后回写本地事实

推荐目录

workspace/
  project-frontend/
  project-backend/
  project-harness/   # 唯一同步中心，可是远程 Git 仓库
    global/
      dsl/            # 共享 DSL 基线
      plans/          # 协作 Plan 基线
        TASK_BOARD.md # 共享任务看板
      contracts/      # 共享接口契约
    projects/
      frontend/       # 前端快照
      backend/        # 后端快照

共享基线与本地派生

共享 DSL 基线project-harness/global/dsl/前后端共同事实：需求、页面/模块、交互、事件、接口边界、权限、错误码、验收、非目标。

同步事件类型

事件类型什么时候使用对方如何处理
complete一个任务完成，可能解除对方依赖或需要对方联调。sync inbox 查看证据，必要时更新本地 Plan / ATOMIC。
change需求、字段、权限、状态、验收标准变化。用 aw req change 采纳到本地，再回写 DSL / Plan / ATOMIC。
contract接口契约、DTO、枚举、错误码、分页规则变化。更新 API client / Service Catalog / 测试用例，并重新 brief 相关任务。
question不确定需求、接口、页面、交互、权限边界。先回答并发布新的 decision 或 handoff，不能让对方猜。
block当前任务被对方接口、字段、页面或决策阻塞。对方优先处理阻塞，或给出替代方案和预计时间。
bug跨端 Bug、联调失败或疑似契约不一致。本地写入 Bug 记录，修复后用 complete 或 contract 回传。
decision跨端共同结论，例如错误码、权限策略、字段默认值。双方都更新本地 DSL / Plan / 代码约束。
handoff普通交接、进度说明、证据路径。只读查看，必要时转成任务、Bug 或需求变更。

19.15c 通用前后端同步事件

所有影响其他 Agent 的事件都使用 aw sync event：

complete  任务完成，会解除对方依赖
change    需求、字段、权限、状态、验收变化
block     当前任务被对方阻塞
question  需要对方确认
contract  接口契约、DTO、枚举、错误码变化
bug       跨端 Bug 或疑似 Bug
decision  跨端共同决策
handoff   普通交接

示例：
./scripts/aw sync event \
  --type contract \
  --task AT-BE-003 \
  --to public-frontend-agent \
  --summary "用户列表接口响应新增 avatarUrl" \
  --impact "影响前台用户列表展示" \
  --acceptance "前端 API client 和列表头像展示完成" \
  --risk "前端不采纳会导致联调字段不一致" \
  --evidence "docs/SERVICE_CATALOG.md; project-harness/global/contracts/"

对方采纳前先执行：
./scripts/aw sync pull --from <peer>

20. FAQ

这些动作会自动后台执行吗？

不会。它不是后台服务。Codex 在执行 AgentWorkflow 流程时可以自动调用命令，但必须在当前任务回合中执行。

同步中心在哪里？

本地同一台电脑时，建议与前后端项目平级：workspace/project-harness。两台电脑时，建议做成单独远程 Git 仓库。

pull 会覆盖本项目文件吗？

不会。它只写入 docs/sync/inbox/<peer>/，作为只读收件箱。

什么时候需要工程师确认？

新需求、研发中变更、接口字段、权限、金额、数据口径、删除/重命名、数据库变更、新依赖、Git 提交、发布都需要确认。

如果 Agent 忘记同步怎么办？

工程师可以要求它执行：aw sync status、aw sync pull --from <peer>、aw sync inbox --from <peer>、aw sync board。如果是跨端事项，要求它补一条 aw sync event，不要只补普通 handoff。

本手册为本地静态 HTML，可直接用浏览器打开。建议与 agent-workflow/INVOCATION.md、CROSS_PROJECT_SYNC.md、skill/reference.md 一起阅读。
